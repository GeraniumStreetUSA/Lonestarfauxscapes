# LSFS Repo Audit

Date: 2026-03-05

Scope: repo inspection only. No production page files were modified as part of this audit.

## Audit Snapshot

- Framework: Vite multi-page static site, not a React/Next/Astro app. Root `*.html` files are the main source templates. `index.tsx` exists but does not appear to be part of the active build.
- Routing model: static `.html` output with Cloudflare Pages middleware canonicalizing to extensionless routes like `/pricing` instead of `/pricing.html`.
- CMS / content source:
  - Decap CMS at `/admin/` manages `content/blog/*.md` only.
  - Case studies are markdown-driven from `content/case-studies/*.md`, but are generated by script rather than managed in Decap CMS config.
  - `content/locations/**` exists, but I did not find an active build path using it.
  - Core money pages are hand-authored root HTML files.
- Build pipeline: `npm run build` runs `scripts/build-blog.js`, `scripts/build-case-studies.js`, `scripts/generate-sitemap.js`, `vite build`, then `scripts/post-build.js`.
- Deploy config: Cloudflare Pages + Pages Functions. Redirects and headers are controlled by `_redirects`, `_headers`, and `functions/_middleware.js`.
- Analytics: no GA4, GTM, Plausible, Clarity, PostHog, Mixpanel, Segment, or similar tracking code was detected in the repo scan.
- Important cleanup finding: current `dist/` output includes `lighthouse-report.html` and `navbar-universal.html`, so those pages are currently build-emitted and likely public unless blocked upstream.

## Source Of Truth Rules

- Edit root HTML files for core landing pages:
  - `index.html`

  - `living-wall.html`
  - `residential.html`

  - city pages like `austin.html`, `dallas.html`, `houston.html`

  - blog hub template: `blog.html`

  - generator: `scripts/build-case-studies.js`

## Risks / unknowns

- The repo is currently dirty. There are many existing modified and untracked files, so content ownership and production readiness of those changes are not fully clear from this audit alone.
- `dist/` confirms current build-emitted pages, but the deployed production environment could still differ if Cloudflare is serving an older build.
- `content/locations/**` exists but is not wired into the active build. It may be planned future work or abandoned work.
- `content/posts.json` is generated and copied to `dist`, but I did not find clear evidence that the live frontend consumes it directly. It may exist mainly for build support / slug preservation.
- `contact.html` contains a fallback submit target on an external worker domain. That means live form behavior may partly depend on infrastructure outside this repo.
- `functions/hedge-quote.js` handles quote submissions, but unlike `/api/contact`, it does not appear to verify Turnstile. That may be intentional, but it is a conversion/security difference worth confirming.
- `pricing.html` currently uses `Article` schema. That may be intentional, but it looks mismatched if the page is meant to be a sales/conversion page.
- `navbar-universal.html` is excluded from sitemap generation but is still emitted in current build output. That suggests a mismatch between "should be public" and "is being built."
- `_headers` contains `/sample-page/feed/`, which looks like a leftover path from an older site or platform and may be harmless clutter, but it is still a sign of config drift.
- No analytics code was detected in the repo. It is possible analytics are injected outside the repo by Cloudflare tooling or another upstream layer, but I could not confirm that from repo contents alone.

## Concise Summary

- The site is a Vite-built static MPA on Cloudflare Pages with extensionless canonical routes enforced by edge middleware.
- Most important revenue pages are hand-authored root HTML files. Blog and case studies are the only real content pipelines, and only the blog is CMS-managed.
- There is no formal shared component system. Reuse is mostly duplicated markup or generator-owned templates.
- The safest edit order is:
  1. technical cleanup
  2. SEO structure
  3. homepage rewrite
  4. pricing rewrite
  5. CRO/form cleanup
  6. case studies/local proof
  7. blog/internal linking
  8. B2B resource hub
  9. analytics/QA
- Biggest cleanup risk before rewriting copy: current build output still includes internal-looking pages like `/lighthouse-report` and `/navbar-universal`.
